Go-TaskQ

A production-style distributed task queue built in Go. It exposes a gRPC API for job submission and polling, stores jobs durably in Redis, supports priority scheduling, delayed retries, dead-letter queues, and includes Prometheus metrics for observability.

View on GitHub

Why I Built This

I wanted to understand how real systems like Celery, Sidekiq, and SQS-based workers behave under failure: retries, backoff, durability, fairness across queues, and what “reliable background processing” actually means when a process crashes or a network hiccups. Go-TaskQ is my attempt to build that foundation end-to-end.

What It Enables

Go-TaskQ is designed for reliable background work such as webhooks, async pipelines, scheduled jobs, and any task you don’t want blocking a request path. The system prioritizes durability, predictable retries, and visibility into what’s happening via metrics.

Key Features

  • gRPC API: Submit jobs and poll for work with a structured, strongly-typed interface.
  • Durable Redis storage: Job metadata is persisted so jobs aren’t lost on restarts.
  • Priority queues: High / medium / low queues to control execution order and fairness.
  • Delayed retries: Retry scheduling using Redis sorted sets (time-based release).
  • Dead-letter queue: Failed jobs move to DLQ after retry exhaustion for later inspection.
  • Leader-aware scheduler: Foundation for HA retry scheduling and coordination.
  • Observability: Prometheus metrics for enqueue throughput, retry movement, and queue depth.
  • grpcurl-friendly: Reflection enabled for fast local testing without writing a client.

Run Locally

Start Redis:

brew services start redis
# or
redis-server

Run the API server:

REDIS_ADDR=localhost:6379 GRPC_ADDR=:9000 go run ./cmd/api

You should see logs like:

[api] gRPC on :9000 (redis=localhost:6379)
[api] metrics on :9100

Test with grpcurl

List services:

grpcurl -plaintext 127.0.0.1:9000 list

List methods:

grpcurl -plaintext 127.0.0.1:9000 list taskq.v1.TaskQ

Enqueue an example webhook job:

grpcurl -plaintext -d '{
  "type":"webhook",
  "payloadJson":"eyJ1cmwiOiJodHRwczovL2h0dHBiaW4ub3JnL3Bvc3QifQ==",
  "priority":"high"
}' 127.0.0.1:9000 taskq.v1.TaskQ/Enqueue

Prometheus Monitoring

Metrics endpoint:

http://localhost:9100/metrics

Run Prometheus locally:

docker run --rm -p 9090:9090 \
  -v "$(pwd)/deployments/prometheus.yml:/etc/prometheus/prometheus.yml:ro" \
  prom/prometheus

Open the Prometheus UI:

http://localhost:9090

Example metrics to explore:

taskq_jobs_enqueued_total
taskq_scheduler_moved_total

What This Project Shows

Go-TaskQ demonstrates core distributed systems patterns: durable job storage, priority scheduling, time-based retries, DLQ mechanics, API design with gRPC, and real observability with metrics. It’s a strong “systems” project that reflects how background processing is built in production.
